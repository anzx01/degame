## 《桌面整理大师》程序实现拆解清单

版本：V1.0  
日期：2026-04-25  
适用范围：客户端开发、技术评审、排期拆解、原型实现

---

### 1. 文档目标

本清单用于把 PRD 落成实际开发任务，重点解决三件事：

* 核心三消循环怎么实现
* “桌面整理感”如何通过系统落地
* 如何在纯 2D Web 项目中控制复杂度和风险

---

### 2. 技术实现原则

#### 2.1 总原则

* 逻辑层与表现层分离
* 棋盘状态可序列化
* 所有动效由事件驱动
* 先保证规则正确，再叠加表现
* 功能拆分必须服务 MVP，不提前做扩展架构

#### 2.2 推荐技术方向

可选实现路线：

* `Canvas/WebGL 2D` 游戏框架路线
* `DOM + Canvas 混合` 路线

建议结论：

* 棋盘、粒子、特效采用 2D 渲染层
* HUD 和菜单可用常规 UI 层实现
* 不建议用纯 DOM 承载高频棋盘动画

### 4. 场景与页面结构

建议最小场景拆分：

* Boot：初始化、资源检查
* MainMenu：主菜单
* ChapterSelect：章节选择
* Gameplay：核心关卡
* Result：结算页
* Settings：设置弹层

要求：

* 场景切换要轻量，不重复销毁公共资源。
* BGM、用户设置、存档服务应在场景间保持可用。

### 6. 棋盘逻辑拆解

#### 6.1 基础流程

一次有效操作的标准流程：

1. 接收玩家交换输入
2. 检查是否相邻
3. 临时交换
4. 扫描匹配
5. 若无匹配则回滚
6. 若有匹配则提交交换
7. 标记消除集合
8. 结算障碍影响
9. 生成特殊工具
10. 清空棋子
11. 处理掉落补位
12. 检测连锁
13. 更新任务与整洁度
14. 判断关卡完成

#### 6.2 匹配扫描

要求：

* 同时扫描横向和纵向
* 支持识别 3 连、4 连、5 连、L/T 型
* 扫描结果必须归一化，避免重复计算同一格

建议输出：

```ts
type MatchGroup = {
  cells: Array<{ row: number; col: number }>
  pattern: "line3" | "line4" | "line5" | "t_shape" | "l_shape"
  tileType: string
}
```

#### 6.3 特殊工具生成

规则建议：

* 4 连生成行/列清除工具
* 5 连生成全盘搜索工具
* L/T 型生成范围清扫工具

注意：

* 工具生成位置应稳定且可预期
* 优先在玩家主动交换涉及的关键格生成

#### 6.4 无解洗牌

要求：

* 只在无可行步时触发
* 洗牌后必须重新验证至少存在一步
* 洗牌尽量保持当前障碍和目标布局

### 7. 障碍系统实现

#### 7.1 障碍抽象

建议统一接口：

```ts
type BlockerState = {
  type: "dust" | "sticky_note" | "popup"
  hp: number
}
```

#### 7.2 触发规则

* 灰尘：相邻消除触发一次清除
* 便签：所在格参与一次消除后清除
* 弹窗：相邻命中两次后移除

#### 7.3 实现注意点

* 障碍结算不要写死在渲染层。
* 每种障碍都必须有独立单测。
* 障碍清除要能发出标准事件给 HUD、动效和任务系统。

---

### 8. 整洁度系统实现

#### 8.1 设计目标

整洁度系统负责把“消除事件”转化为“桌面恢复进度”，这是题材闭环的核心。

#### 8.2 计算建议

整洁度来源：

* 基础消除：小幅增加
* 障碍清除：中幅增加
* 工具触发：按影响范围额外加成
* 章节关键障碍：可设置固定节点奖励

建议：

* 整洁度计算不等于分数。
* 整洁度应服务关卡目标和背景恢复，而不是纯奖励数值。

#### 8.3 节点事件

建议在以下节点发事件：

* `25%`
* `50%`
* `75%`
* `100%`

用于：

* 背景图层切换
* 音效增强
* HUD 提示

### 9. 任务系统实现

#### 9.1 任务模型

```ts
type GoalConfig =
  | { type: "collect_tile"; tileType: string; target: number }
  | { type: "clear_blocker"; blockerType: string; target: number }
  | { type: "reach_cleanliness"; target: number }
```

#### 9.2 更新时机

任务应在以下节点更新：

* 普通消除提交后
* 障碍清除完成后
* 整洁度变化后

#### 9.3 判定原则

* 任务判定只依赖逻辑状态
* 不依赖动画播放结束
* 结算条件由统一规则层触发

---

### 10. 输入与状态控制

#### 10.1 输入控制

需要处理：

* 点击选中
* 拖动交换
* 移动端触摸滑动
* 播放动画期间输入锁定

#### 10.2 状态机建议

```ts
type GamePhase =
  | "idle"
  | "swapping"
  | "resolving"
  | "dropping"
  | "checking"
  | "result"
  | "paused"
```

原则：

* 任一时刻只允许一个主状态生效
* 输入是否可用由状态机统一判断

---

### 11. 表现层对接

#### 11.1 事件驱动结构

逻辑层应抛出标准事件，例如：

* `swap_started`
* `swap_reverted`
* `match_created`
* `tiles_cleared`
* `blocker_damaged`
* `blocker_removed`
* `special_spawned`
* `special_triggered`
* `cleanliness_changed`
* `level_completed`

#### 11.2 对接原则

* 动效和音效只消费事件，不反向修改规则。
* HUD 只读取状态，不参与逻辑结算。

### 12. 存档与配置

#### 12.1 本地存档字段

建议保存：

* 当前解锁章节
* 每关星级
* 每关最佳步数
* 教学是否完成
* 音量和震动设置

#### 12.2 存档要求

* 场景切换时自动保存
* 关键完成节点强制保存
* 版本升级要预留迁移字段

### 13. 埋点与调试

#### 13.1 最小埋点

* 首局开始
* 首局完成
* 第 1/5/10 关完成
* 使用提示次数
* 单局步数
* 单局时长
* 退出关卡时所在关卡

#### 13.2 调试工具

建议提供开发开关：

* 指定关卡直达
* 强制生成特殊工具
* 查看所有可交换位置
* 跳过结算
* 打印匹配结果

### 14. 性能策略

#### 14.1 渲染

* 高频元素使用图集
* 背景分包加载
* 粒子数量按设备降档

#### 14.2 逻辑

* 匹配扫描在棋盘维度内完成，不做不必要全局重算
* 只对受影响区域二次检查时可做局部优化

#### 14.3 内存

* 场景切换清理临时特效对象
* 控制动画实例复用

### 15. 开发顺序建议

#### 阶段 1：最小可玩原型

* 8x8 棋盘
* 基础交换
* 匹配消除
* 掉落补位
* 无解洗牌

#### 阶段 2：题材闭环原型

* 任务系统

* 3 种障碍

* 引导和存档

#### 阶段 4：打磨

* 动效
* 音效

* 性能优化

### 16. 风险点与规避

| 风险 | 说明 | 规避方式 |
| ---- | ---- | -------- |
| 逻辑和表现耦合 | 改动动效时容易破坏规则 | 事件驱动，逻辑层独立 |
| 关卡数据混乱 | 后续内容扩展困难 | 统一配置结构和校验 |
| 整洁度沦为假进度条 | 题材价值感消失 | 背景恢复与结算对比强绑定 |
| 障碍过多过杂 | 规则理解负担上升 | MVP 只做 3 类障碍 |
| 动效过慢 | 消除节奏拖沓 | 高频动效统一时长上限 |

---

### 17. 交付验收标准

达到以下标准，才算程序实现满足 PRD：

* 规则稳定，无异常死盘。
* 任意有效交换都可正确结算。
* 特殊工具生成和触发可预测。
* 障碍规则与任务规则一致。
* 整洁度能真实反映局内推进。
* 关卡流程从进入到结算完整闭环。
* 低端设备上仍可稳定游玩。

---

### 18. 一句话标准

高质量程序实现的标准是：**规则层稳定、表现层清晰、题材闭环成立，并且整个项目没有为未来想象中的复杂版本提前付出过高成本。**
